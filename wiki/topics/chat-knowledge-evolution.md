# Chat knowledge source mimari evrimi

> **TL;DR:** Chat'in "haber kaynağı yetersizse genel bilgi getir" yeteneği tek seansta **5 mimari iterasyon** geçirdi. Confidence-routing → CTA banner → pattern-matching → fast-path hepsi production'da kırıldı veya kullanıcı tarafından reddedildi. Doğru çözüm en başta basitti: **LLM tool-use** (LLM kaynak yetersizse kendi karar verir). Bu sayfa neyin neden başarısız olduğunun anti-pattern kataloğudur — [[failed-experiments-rag-quality]] benzeri.

## Bağlam

Faz 1 chat-only migration (#800-#807) sonrası kullanıcı sohbeti "general assistant" gibi kullanmaya başladı ("trump kaç yaşında", "stargate atlantis kaç sezondu"). Sistem haber arşivinde arayıp "kaynaklarda yok" diyordu. Çözüm: haber yetersizse Wikipedia/Wikidata'ya başvur. Ama "nasıl?" sorusu 5 iterasyon sürdü.

## Çıkarılan dersler (anti-pattern listesi)

1. **Retrieval skoru ≠ "cevap var mı".** RRF/semantic skoru konunun *geçtiğini* ölçer, *cevaplandığını* değil. Confidence-based routing bu yüzden temelden kusurlu (#810).
2. **LLM çıktısını pattern-match etme.** "Kaynakta yok" cevabını regex'le yakalamak brittle — LLM her zaman farklı cümle kurabilir (#819, kullanıcı reddetti).
3. **Planner-generated query Wikipedia için kötü.** topic_query haber-retrieval'e optimize (genişletilmiş, soru kelimeli). Wikipedia entity araması ister; LLM'in ürettiği temiz entity ("Yıldız Geçidi Atlantis") doğru (#826 fast-path revert).
4. **Karar verici LLM olmalı, heuristik değil.** Kaynakları gören LLM "yeter mi" kararını verir — planner accuracy'sine veya skora bağımlı değil. Bu güvenlik ağı.
5. **Latency optimizasyonu doğruluğu bozmamalı.** fast-path ~3.7s kazandırıyordu ama yanlış sayfa getiriyordu. Doğruluk öncelikli.
6. **REST summary ≠ structured data.** Wikipedia extract prose'dur, infobox (doğum/nüfus) ayrı katman → Wikidata gerekli.
7. **Latent bug tüm mimariyi maskeler.** #820 stream bug Faz 1'den beri vardı; fallback path her zaman çalıştığı için Faz 2 mimarisi gerçekte hiç test edilmemişti. Yeni mimari eklerken alt-katman sağlığı doğrulanmalı.
8. **Conversational retrieval = ayrı condense adımı.** Follow-up bağlamını planner'ın sabit prompt'una gömmek çalışmaz (system prompt baskın, #832). Standalone query üretimi izole bir LLM call olmalı (#833 — sektör standardı).
9. **Bağlam tüm pipeline'a tutarlı akmalı.** effective_query sadece planner'a değil retrieval + tool query + gen_user_msg'e de gitmeli; bir yerde HAM mesaj kalırsa o noktada bağlam kopar (#829 retrieval, #835 tool query).
10. **Tool-decision streaming provider-bağımlı — DeepSeek streaming+tools BROKEN.** #836'da "generate_text_stream(tools=) + final-chunk tool_calls ile gerçek streaming korunur" varsayıldı; DeepSeek'te yanlış çıktı: streaming+tools yapısal `delta.tool_calls` vermez, `<｜DSML｜tool_calls>` özel token'ını content'e ham XML basar (kullanıcı ham token görür + uzun-yazıp-kısaya-dönme). Doğru desen (#840): Aşama 1 **non-streaming** `generate_text(tools=)` (yapısal tool_calls), content yield etme; tool varsa Aşama 2 **TOOLSUZ** `generate_text_stream` (DSML yok → gerçek streaming); tool yoksa `_simulate_stream` (decision_text zaten üretildi, ekstra LLM call yok). Ders: streaming+tool-call davranışı provider-spesifik, OpenAI formatı varsayma; mevcut benchmark/test yeni provider davranışını doğrulamadan "streaming korunur" deme.
11. **Tool query = kanonik entity, soru/niteleyici DEĞİL.** LLM `search_wikipedia`'ya "Stargate SG-1 4. sezon" gönderince TR Wikipedia full-text alakasız sayfa ("200/Paul Mullie") döndü; temiz "Yıldız Geçidi SG-1" → #1 doğru ana sayfa (canlı API kanıtı). Wikipedia entity araması ister; niteleyici (sezon/bölüm/tarih) relevance'ı kirletir. Anti-pattern #3'ün (planner query Wikipedia için kötü) LLM-tool-query'ye genişlemesi — tool param + prompt entity-only zorlamalı.
12. **Kaynak sub-fact'i içermiyorsa LLM uydurur + sahte citation ekler (C1 en tehlikeli kaçak).** "S4E1 adı" hiçbir REST özetinde/Wikidata'da yoktu; LLM cevabı kendi belleğinden üretip [W1] (alakasız sayfa) iliştirdi — doğru cevap ama kaynaksız, brand güvencesi (C1) delindi. 31. **Bir bağlamı LLM'e GETİRMEK yetmez — onu NASIL kullanacağını da söyle; "context var ama talimat dar" sessiz bir kusurdur.** (#955, conv 9dc4b0b0; #888 ailesi devamı) #888 sohbet hafızasını answer LLM'e taşıdı (`_recent_conversation_context` koşulsuz) ama followup_block talimatı yalnız "olgularla tutarlı ol / çelişkiyi düzelt" idi. Sonuç: kimlik/selamlama her turda birebir tekrar, haber follow-up'ı baştan tekrar — LLM context'i görüyor ama "tekrarlama, üzerine ekle, akıcı devam et" denmediği için system-prompt kalıbına düşüyor. Genel dersler: (a) **veri-yolu fix'i (context getir) + kullanım-talimatı (context'le ne yap) AYRI işlerdir** — #888 ilkini çözdü, ikincisi eksik kaldı; bir bağlam enjekte ettiğinde "LLM bununla ne yapmalı?" talimatını da AYNI anda ver (yoksa context israf + sessiz davranış kusuru); (b) **prompt kuralları durum-duyarlı olmalı** — "kimlik sorulursa tanıt" kuralı konuşma-durumunu (ilk-temas mı, sürüyor mu) hesaba katmazsa her turda ezber tekrar; "X sorulursa Y yap" kurallarına daima "konuşma bağlamı şuysa farklı yap" istisnası düşün; (c) kullanıcının "akıcılık kaybı" gibi stilistik şikayetleri de evergreen kök gerektirir (tek conv'a patch değil) — burada kök "context kullanım talimatı eksik", çözüm prompt-katmanı ama YAPISAL (followup_block + system prompt, tek-örnek değil); (d) stilistik/akıcılık sorunları olgusal değil → #906 "deterministik kod-sinyali" deseni UYGULANMAZ; doğru katman prompt (ama yapısal, conv-agnostik). #888 ders #24 ile aynı omurga: "ilgili-mi (retrieval) ≠ hafıza (her zaman)" → şimdi "hafıza var (context) ≠ hafızayı doğru kullan (talimat)".

32. **LLM saran ürünü/sistemi eğitim verisinden BİLMEZ; self-knowledge kanonik prompt'a yazılmalı + tool'suz meta-path'e C1 backstop; "küçük statik bilgi = prompt (caching), tool DEĞİL".** (#958, conv b107069a) Sistem "neden adın Nodrat" sorusuna "Taylor Swift tersten" diye asılsız etimoloji uydurdu. Genel dersler: (a) **bir LLM, içine gömüldüğü ürünün kimliğini/kökenini doğal olarak bilmez** (eğitim cut-off / niş) — "model nasılsa bilir" varsayma; kimlik kanonik olarak prompt'a yazılmazsa spesifik kimlik-sorusu = garanti halüsinasyon; (b) **C1 (kaynaksız iddia yasağı) yalnız tool-çağıran path'lerde değil, tool'suz meta/kimlik path'inde de gerekir** — tool yoksa doğrulayacak kaynak da yok, bu yüzden "bilmiyorsan uydurma, kanonik dışına çıkma" backstop'u tam da orada şart (en çok unutulan yer); (c) **maliyet sezgisi yanıltıcı: "system prompt'u genişletmek pahalı" → prefix-caching ile statik prompt cache-hit'te ~10× ucuz; küçük kanonik bloğun marjinal maliyeti ≈0**; bunu "ayrı tool'a bağlayalım" demek schema-token + ekstra round-trip + hata yüzeyi ekler (küçük/statik bilgi için tool = anti-pattern). Tool'un eşiği: bilgi **büyük + dinamik + sık değişen** (detaylı SSS/fiyat/sürüm); o zaman self-docs retrieval; (d) referans mimari (Perplexity): model-agnostik orkestrasyon her isteğe kurumsal kimliği **prompt-enjekte** eder (model eğitiminden değil) + detay için kendi docs-retrieval'ı = **hibrit**; "her model X'i biliyor" çünkü prompt enjekte ediliyor, model bilmiyor. Karar [[self-identity-canonical-prompt]].

33. **Yeni özelliği mevcut omurgaya/kimliğe SAYGIYLA ekle: ayrı izole call > invaziv yeniden-yapı; ürün-kimlik gerilimini özelliğin kapsamını daraltarak çöz.** (#961, Perplexity-parite takip soruları) Kullanıcı 2 davranış istedi (cevap-içi "istersen" + 5 takip sorusu). 34. **Mutlak tarih-atfı (absolute temporal grounding) ≠ olaylar-arası ilişkisel/nedensel çıkarım (relational temporal reasoning); tarihleri doğru söylemek yetmez, soru sıralama/nedensellik içeriyorsa karşılaştır + sonucu açıkça kur.** (#964, conv 'Bugünkü gündem' #5) #879/#928 ile sistem her olayın tarihini DOĞRU atfediyordu (Trump 17 May, Erakçi 14 May) ama "X'e yanıt geldi mi?" sorusunda 14<17 → "Erakçi öncesinde → yanıt değil" mantıksal köprüsünü kurmadı; 14 May'i yanıt-adayı gibi sundu. Genel dersler: (a) **bir yetkinliğin (tarih atfı) çözülmüş olması, onun ÜZERİNE kurulu akıl yürütmenin (tarihler-arası ilişki) de çözüldüğü anlamına gelmez** — katmanlı yetenekleri ayrı doğrula ("tarihleri biliyor" ≠ "tarihler-arası mantığı kuruyor"); (b) #879 ders #22 / #888 ders #24 / #955 ders #31 omurgası tekrar: bir veri context'te DOĞRU bulunsa bile onunla NE yapılacağı AYRICA talimatlanmalı — "tarihi atfetmek ≠ tarihler-arası ilişkiyi çıkarmak"; (c) ironik tanı sinyali: #961 takip-üreticisi tam da eksik çıkarımı ("X, Y'den önce mi?") soru olarak üretti → bilgi sistemde VAR, ana cevap kullanmadı; **alt-bileşenin doğru soruyu üretmesi, ana cevabın o çıkarımı kapatması gerektiğinin kanıtı** (cevap takip sorusuna ihtiyaç bırakmamalı); (d) çözüm katmanı ayrımı kritik: tarihler context'te zaten doğruysa kök = reasoning-talimatı eksikliği → prompt; #906 kod-sinyali deseni UYGULANMAZ (orada veri DÜŞÜYORDU, burada değil). 35. **Doğru kaynağı GETİRMEK ≠ doğru kaynağı SEÇMEK; bir retrieval motoru aday kümeyi döndürüyorsa, "hangi aday cevabın temsilcisi" ayrı bir karar — ve bu karar relevance'a körü körüne bırakılamaz.** (#967, conv 3f1ca529, "Yıldız Geçidi SG-1") Wikipedia full-text doğru kanonik maddeyi sonuç kümesine KOYUYORDU ama #1 sıraya değil; `articles[0]` (hem #863 QID hem ilk `[n]` bloğu temsilci) yan sayfa olunca cevap yanlış sayfaya dayandı. Genel dersler: (a) **#964 ders #34'ün retrieval ikizi: "veriyi bulmak ≠ veriyle ne yapacağını bilmek" — burada "doğru sayfa kümede ≠ doğru sayfa temsilci"**; aday üreten her katmandan sonra "temsilci seçimi" explicit tasarlanmalı (örtük "ilk eleman" = gizli relevance'a güven); (b) **çözüm katmanı ayrımı (#906 vs #931/#955 ekseninin üçüncü hali): veri DÜŞMÜYOR (≠#906 kod-sinyali) ve saf-reasoning DEĞİL (≠#964 prompt) — veri kümede ama SIRA yanlış → deterministik seçim/sıralama kodu** (tool-sarmalı politikası #906/#928/#879 ailesi; generic motor `wikipedia.py` DEĞİŞMEZ); prompt değil çünkü LLM'e zaten yanlış sıralı küme gidiyor, talimat değil seçim sorunu; (c) **#863 granülasyon emsali**: aynı aile/karakterdeki onarım (tool spec & query DEĞİŞMEZ, saf veri/seçim-yolu) ayrı decision sayfası DEĞİL, locked kararın callout-evrimi — over-granülasyon kaçınıldı (memory: "mimari karar değişikliği=ayrı sayfa" ama bu karar-değişikliği değil, kararın içinde rafine); (d) **geri-uyum kapısı = riski sınırlama**: heuristik (tam-başlık eşleşmesi) tetiklenmiyorsa eski davranış aynen — yeni kod yalnız yüksek-kesinlikli sinyalde devreye girer, belirsizde mevcut sistemi bozmaz (kullanıcı onaylı muhafazakar default); (e) Türkçe normalize her yerde aynı tuzak: #939 collation dersi (`İ`→`i`, `I`→`ı`, U+0307) string-eşleştirme yapan HER yeni kodda tekrar uygulanır (DB-only değil, Python-side karşılaştırmada da).
